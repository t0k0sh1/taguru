ライブラリ層——1コンテキスト＝1枚のフラットバッファ

context.rs。全状態が1つの平坦なバッファに収まるという不変条件が、この下のすべてを単純にする。

  • 状態は「UTF-8文字列アリーナ＋固定幅 #[repr(C)] レコードの7テーブル」。隣接リストはエッジレコードに縫い込まれた侵入型チェーン（intrusive chains）で、ポインタの追跡が別アロケーションを生まない。
  • すべての変更は「追記」か「フィールド更新」。だから全状態が to_bytes／from_bytes で1枚のイメージとして往復する——リトルエンディアン、ロード時に全検証。
  • 容量はu32空間：テーブルあたり約42.9億レコード、インターン済みテキスト4 GiB。超過は ContextFull エラーであって、パニックではない（HTTPでは507になり、書き込みは適用されていない）。
  • 検索の入口は正規化（NFKC・大小・カナ→ひらがな）＋バイグラム転置索引。エイリアスは入口専用の別綴りで、結果は常に正準の綴りを運ぶ。

サーバー層——レジストリとファイルファミリー

registry.rs。ディスクが真実の源で、メモリはコンテキスト丸ごと単位のキャッシュ。

data/
├── {name}.ctx                 # グラフのイメージ（to_bytesの生バイト）
├── {name}.meta.json           # 説明・ピン留め・統計のサイドカー（イメージを純粋なダンプに保つため外に置く）
├── {name}.wal.jsonl           # グラフの書き込みWAL（flush成功のたびに切り詰め）
├── {name}.passages.bin        # 原文のコンパクション済みスナップショット
├── {name}.passages.wal.jsonl  # 原文の追記ログ
├── {name}.bm25.bin            # 段落転置索引（派生物——失っても再構築、障害にはならない）
├── {name}.vectors.bin         # 埋め込みキャッシュ（モデル名がキー）
├── oauth.json                 # リモートMCPのOAuthグラント
└── .taguru.lock               # データディレクトリの advisory ロック
  • ブートは全コンテキストをコールド登録（ピン留めだけ先読み）。最初のアクセスが透過的にロードし、キャッシュ予算（TAGURU_CACHE_BYTES）を超えるとLRUで退避。
  • 書き込みはコンテキストをdirtyにするだけ。永続化は周期flusher（TAGURU_FLUSH_SECS）、退避時、シャットダウン時。
  • 1コンテキストの真実はファミリー一式——バックアップは必ず一括（運用の基本）。

耐久性——WALが「200＝残る」を作る

wal.rs・passages.rs。flush間隔は鮮度の刻みであって、損失窓ではない。

  • HTTP APIが受理したグラフ変更は、メモリに触れる前にWALへ追記される（JSON Lines、バッチごとにfsync）。クラッシュしても、次のブートがリプレイして戻す。
  • リプレイはシーケンス番号だけで駆動される：各レコードが単調な seq を持ち、イメージのヘッダが「ここまで焼き込み済み」の透かし（watermark）を持つ。二重適用も取りこぼしも構造的に起きない。
  • WALはflush成功のたびに切り詰められる。上限（既定256 MiB）に近づくのはflushが失敗し続けているときだけで、超えたら書き込みを500で拒否する——ログを無限に伸ばすより、書けない事実を早く言う。
  • 原文は独立した耐久経路を持つ：コンパクション済みスナップショット＋常時ONの追記ログ。前身は1ファイルを毎回全書き換えで、N文書のimportがO(N²)バイトを書いていた——今はグラフと同じ「スナップショット＋ログ」の形。
  • すべてのライターがfsync＋rename。ファイルシステムのスナップショットはいつの瞬間でも安全で、taguru inspect がサーバーと同じ全検証ロード＋WALリプレイでバックアップを裏取りする。

三つの検索経路

グラフの入口、字句の網、意味の網。どれが答えたかは常に結果が申告する。

グラフの入口

正規化＋バイグラム索引でcueを概念に解決し、あとは構造を手繰る（describe／query／activate／explore）。類似度はここに登場しない。

字句レーン（BM25）

段落の転置索引を常駐させ、クエリごとの再トークン化を止める。原文ストアが真実で、索引は派生物——欠けていれば再構築される。更新はソース単位の墓標＋追記で増分。

セマンティックレーン（任意）

概念・ラベルはグロス（名前＋最重量の事実）として埋め込む——単語だけでは文で訓練されたモデルに信号が足りない。TAGURU_EMBED_PASSAGES で段落とdoc2query質問も対象に。

  • 「段落」を決める関数はひとつ（paragraph.rs）：BM25も、段落埋め込みも、原文ストアの区間も同じ分割を輸入する。三者が食い違う余地を最初から作らない。分割は機械的——空行の連なりが境界、正規化なし。
  • 両レーンが走ったときの sources/search のスコアはランク融合、片方ならその素点。ヒットの lanes がレーンごとの順位と素点を必ず添える。
  • 埋め込みリクエストには X-Taguru-Embed-Purpose: index | query が付く——非対称モデル（Cohere・Voyage）を前置きプロキシで正しく振り分けるためのフック。素のOpenAIサーバーは無視する。
  • ベクトルは常駐で、上限は TAGURU_PASSAGE_VECTOR_LIMIT（既定20,000行）。超えた分は字句レーンが全段落を守り、セマンティック側だけが部分的になる（refreshの応答がスキップを報告する）。

MCPの表面はひとつ

mcp.rs・remote_mcp.rs・src/bin/taguru-mcp.rs。

ツール定義、ツール→HTTPリクエストの対応、JSON-RPCの枠組みは mcp.rs に一度だけ書かれ、 stdioブリッジと POST /mcp が共有する。トランスポートの違いは「翻訳したリクエストをどう実行するか」 （ブリッジはureqの往復、サーバー内は同一プロセスのRouter呼び出し）と「返事をどう運ぶか」だけ。 リモート側はステートレス・プロファイル：initialize は規律書を配り、各ツール呼び出しは自立している—— SSEもセッションIDも、再開すべきものが無い。

認証・認可・上限

auth.rs・oauth.rs・limits.rs。

  • Bearerは名前付きキーリング：TAGURU_API_TOKEN（キー名 "default"）と TAGURU_API_TOKENS（"ci:tokA,laptop:tokB"）。アクセスログがどのキーかを言い、失効は1本ずつ、ローテーションは重ね掛け（新キー追加→移行→旧キー削除）。不正な設定は起動拒否。
  • OAuthはリモートMCPのため：RFC 9728のリソースメタデータ＋内蔵の最小認可サーバー（PKCE・動的登録込み）。「ログイン」は既存APIキーの所持そのもの——同意ページにキーを貼ると、その接続がそのキーとして振る舞う。トークンはSHA-256でハッシュ化して保存、グラントは data_dir/oauth.json（削除＋再起動で失効、30日未使用で自動失効）。OAuthが開くのは /mcp だけ。
  • 上限は層で分担：リクエスト全体の時間予算とキー毎レートは limits.rs（1分ぶんバーストしてから定常レートに収束、超過は Retry-After 付き429）。ボディ上限は本体の層、パラメータ上限は各ハンドラ（連想バッチ10,000／リスト入力1,000／limit は全所1,000天井）。
  • ブルートフォース対策：失敗したBearer試行はソースIPごとに毎分制限（既定10）。正しいトークンは絶対に絞られない。X-Forwarded-For は信用しない——リバースプロキシの背後では全員がプロキシのIPを共有するので、そこではプロキシ側で絞る。

可観測性

metrics.rs・trace.rs。手書きのRED——数個のatomicと1つのrender関数で足りるものにファサードcrateは要らない。

  • GET /metrics：ルート別のリクエスト数とレイテンシヒストグラム、キャッシュ/flush/WAL/埋め込みの結果、500の原因内訳（taguru_errors_total{kind=…}）、検索のヒット/空振り（taguru_searches_total）、resolveのtier内訳（taguru_resolves_total——semanticの比率が上がる＝cueが語彙から漂流中）、WAL・原文ログのサイズゲージ、最終flush成功時刻。
  • コンテキスト名はメトリクスラベルに入れない：名前はクライアントが鋳造するので、入れた瞬間に系列が無限に増える。コンテキスト別の数字は GET /contexts の使用統計（読み・空振り・書き・最終アクセス）が持つ——advisoryで、flushごと＋graceful shutdown時に永続化。読み出しがディスク書き込みを起こすことはない。
  • 分散トレースはopt-in：OTEL_EXPORTER_OTLP_ENDPOINT を置けば全リクエストがOTLPスパンになり、流入する W3C traceparent／AWS X-Amzn-Trace-Id（ALB・API Gateway）のトレースに合流する。埋め込みプロバイダの往復は子スパンで見え、アクセスログに trace_id が載ってログとトレースが相互参照できる。バックエンドはOTLPなら何でも——切り替えはコレクタ設定であってTaguruの変更ではない。

配布物とサイジング

  • Dockerイメージはscratchにバイナリ1枚：muslの静的リンクで〜13 MB。シェルなし、libcなし、CAバンドルなし（TLSルートはバイナリに焼き込み）。HEALTHCHECKは taguru health——バイナリの自己プローブ。uid 65532、--read-only 可。
  • サイジングは測る：taguru estimate --associations 1_000_000 が目標規模のコンテキストを実際に組み立てて実測する。レイテンシは cargo run --release --example benchmark（10万/100万件での操作別計測）——ただしexamplesは現在整理中。

各判断の一次資料はソースのモジュールコメント： registry.rs（ディスクが真実）、wal.rs（seqとwatermark）、passages.rs（O(N²)からの脱出）、 bm25.rs（派生物という立場）、paragraph.rs（唯一の分割関数）、embedding.rs（グロスを埋め込む理由）、 mcp.rs（一度だけ書く）、oauth.rs（内蔵ASの割り切り）、limits.rs・metrics.rs・auth.rs。
