Amazon Bedrockで使う

Taguruはモデル非依存——サーバーはHTTPを、ブリッジはMCP over stdioを話し、 どのモデルがエージェントを駆動しているかを気にしない。このページは3つの統合点と、 実際に全部を動かして踏んだ角を記録する。

エージェント: Claude Sonnet 4.5（us.推論プロファイル） 埋め込み: amazon.titan-embed-text-v2:0 リージョン: us-east-1

1. エージェント側——Converse + taguru-mcp

MCPが通るところはMCPで。通らないところにはHTTP APIが残っている。

Claude CodeをBedrockで使うなら（CLAUDE_CODE_USE_BEDROCK=1）、特別なことは何もない—— MCPサーバーはローカルの子プロセスなので、通常のセットアップの claude mcp add taguru … がそのまま動く。

Converse APIで自前のエージェントループを組むなら、taguru-mcp をstdioの子プロセスとして抱えて機械的に翻訳する：

  • initialize が規律書の全文を instructions として返す（GET /protocol から生きた設定つきで配られる）。これをConverseの system テキストに——毎ターン再送されるので、cachePoint ブロックの後ろに置くのが望ましい。
  • tools/list の各エントリはConverseのツールに1:1で対応する：{"toolSpec": {"name", "description", "inputSchema": {"json": <MCPのinputSchema>}}}——スキーマは素のJSON Schemaで、無加工で通る。
  • stopReason == "tool_use" なら各ブロックを tools/call に転送し、テキストを toolResult として返す。end_turn までループ。

Sonnet 4.5での実測：9つの事実を含む日本語メモが、規律どおりの8ツール呼び出し （create → 一括の add_associations → store_passages → audit_coverage → 検証）で取り込まれた。 同じ文書の再実行は retract-then-reingest のdiff同期になり、weightの二重加算なし—— 規律書だけで、Anthropic以外のホストAPIの上でも規律が運ばれた。

マネージドなエージェント実行環境（Bedrock AgentCoreなど）はリモートのMCPエンドポイントを欲しがる—— それもある：POST /mcp がMCP Streamable HTTPを、他のAPIと同じBearerトークンの内側で話す （TLSはリバースプロキシで）。https://your-host/mcp をそのまま登録する。 stdioブリッジは同居構成向けの正解であり続け、MCPが届かない場所でも、HTTP APIは Bearer認証つきの小さなJSON APIとしてOpenAPIアクショングループ／Gatewayターゲットに収まる。

2. 埋め込み——ブリッジプロキシの背後のBedrockモデル

Taguruが話す埋め込みプロトコルはひとつ：OpenAI互換の POST {model, input} → {data: [{embedding}]}。 Bedrockの埋め込みモデルは InvokeModel + SigV4 の向こうに、モデルファミリーごとのボディ形で座っている。

橋渡しはLiteLLMやAWSのBedrock Access Gatewayサンプル（どちらも既にOpenAIを話す）で—— あるいはこの程度の小ささのプロキシでも足りる（エラー処理は省略。バインドはloopbackだけに）： 

#!/usr/bin/env python3
# OpenAI互換 /embeddings → Bedrock InvokeModel（Titan & Cohere）
import json
from http.server import BaseHTTPRequestHandler, HTTPServer
import boto3

bedrock = boto3.Session(region_name="us-east-1").client("bedrock-runtime")

def invoke(model_id, body):
    response = bedrock.invoke_model(modelId=model_id, body=json.dumps(body))
    return json.loads(response["body"].read())

def embed(model_id, texts, purpose):
    if model_id.startswith("cohere.embed"):
        # X-Taguru-Embed-Purpose はCohereが求める非対称性そのもの
        input_type = "search_document" if purpose == "index" else "search_query"
        out = []
        for i in range(0, len(texts), 96):  # Cohereは96テキストが上限
            out += invoke(model_id, {"texts": texts[i : i + 96],
                                     "input_type": input_type})["embeddings"]
        return out
    shape = (lambda t: {"inputText": t, "dimensions": 512}) if "v2" in model_id \
        else (lambda t: {"inputText": t})  # Titan: 1呼び出し1テキスト
    return [invoke(model_id, shape(t))["embedding"] for t in texts]

class Handler(BaseHTTPRequestHandler):
    def do_POST(self):
        request = json.loads(self.rfile.read(int(self.headers["Content-Length"])))
        texts = request["input"]
        texts = [texts] if isinstance(texts, str) else texts
        purpose = self.headers.get("X-Taguru-Embed-Purpose", "query")
        vectors = embed(request["model"], texts, purpose)
        body = json.dumps({"data":
            [{"embedding": v, "index": i} for i, v in enumerate(vectors)]}).encode()
        self.send_response(200)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)

HTTPServer(("127.0.0.1", 8257), Handler).serve_forever()

Taguruをそこへ向ける：

TAGURU_EMBED_URL=http://127.0.0.1:8257/v1/embeddings
TAGURU_EMBED_MODEL=amazon.titan-embed-text-v2:0
TAGURU_SEMANTIC_FLOOR=0.2   # 下で校正する——飛ばさないこと
TAGURU_EMBED_AUTO=1         # エージェントのrefresh呼び出しは当てにしない

Taguruは受け取ったベクトルを正規化するので、Titanの生（非正規化）出力はそのままでよい。 ただし dimensions: 512 は固定で——次元設定の違うベクトルは絶対に同じサイドカーを共有してはならず、 Taguruのベクトルキャッシュはモデル名だけをキーにする。次元は一度決めたら変えない。

3. TAGURU_SEMANTIC_FLOOR の校正

床は埋め込みモデルの属性。既定値のまま別モデルに乗り換えると、静かに全部を捨てることがある。

内蔵の床（0.35）は text-embedding-3-large のグロス向けに校正されている。 Titan V2（512次元）は日本語のコサインを圧縮する：実測では正解が0.2〜0.3、無関係な名前が〜0.15に並び、 既定の0.35は正しい答えを全部黙って捨てていた——resolveは [] を返すのに、 その下のランキングは完璧、という症状になる。

埋め込みモデルごとに一度だけ校正する：

  1. 代表的な内容をインジェストし、埋め込みをrefreshする。
  2. 格納済みのどの名前とも綴りを共有しない言い換えcueで突く：POST /contexts/{name}/resolve {"cue": "…", "semantic_floor": 0.05}
  3. スコアが2つの帯に分かれて見える。TAGURU_SEMANTIC_FLOOR をその間に置く。
モデル床
text-embedding-3-large0.35（既定値）
amazon.titan-embed-text-v2:0（512d・日本語）〜0.2

4. Bedrockがモデルの提供を拒むとき

サードパーティモデルへのアクセスはばたつくことがある——成功した直後に同一リクエストが拒否される、 をコンソール側の状態が伝播する間くり返す。当て推量は飛ばして、1コールで全ゲートに名前を付ける。 

aws bedrock get-foundation-model-availability \
  --model-id anthropic.claude-sonnet-4-5-20250929-v1:0
  • regionAvailability — そもそもこのリージョンで提供されているか
  • entitlementAvailability
  • authorizationStatus — Anthropicのユースケースフォーム（コンソールで、アカウントごとに一度。個人なら自分の名前とGitHubのURLで足りる）
  • agreementAvailability — AWS Marketplaceの契約

コンソールからの修復に抵抗するのは agreementAvailability: NOT_AVAILABLE （呼び出しエラーはIAMのせいにし続けるのに、コンソールには保留中のものが何も見えない）。契約は直接作る： 

aws bedrock list-foundation-model-agreement-offers --model-id <id>   # → offerToken
aws bedrock create-foundation-model-agreement --model-id <id> --offer-token <token>

Bedrockのモデルオファーは前払い$0の従量制。契約の作成には呼び出し側のアイデンティティに aws-marketplace:Subscribe + aws-marketplace:ViewSubscriptions が一度だけ要る—— 成功したらそのグラントは剥がしてよく、以後の普通の呼び出しは bedrock:InvokeModel だけで済む。

あと2つの罠：

  • 新しめのClaudeモデルは素のモデルIDを拒む（"on-demand throughput isn't supported"）：クロスリージョン推論プロファイル（us.／apac.／global. プレフィックス）を呼び、IAMは推論プロファイルのARNと、ルーティングされ得る全リージョンの基盤モデルARNの両方に付与する——リージョンはワイルドカードで。
  • Amazonの自社モデル（Titan）はMarketplaceを通らない。Titanは動くのにAnthropic/Cohereが失敗するなら、それは契約ゲートであって、あなたのポリシーではない。
