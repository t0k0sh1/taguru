コンセプト——連想、文脈、二本のレーン

Taguruが何を保持し、どう探し、何をあえてグラフに書かないか。 この設計を知っていると、インジェストの質がそのまま検索の質になる理由が見える。

知識は連想である

最小単位は (subject, label, object, weight, source) の五つ組。

weightは符号付きの証拠量。負の値は否定の主張になる——「大量生産を行わない」は (青嶺酒造, 行う, 大量生産, −1.0)。 否定は肯定と同じラベルに負の符号で載るから、読み違えようがない。

再主張は加算、出典は保存。同じ三つ組を別の文書が主張するとweightが積み上がり、 ソースごとのattributionが全部残る。2ソース × 1.0（独立した裏付け）と 1ソース × 2.0（ひとつの強い主張）は最後まで区別できる。 だから同一文書内での言い換えの再主張は禁物（証拠の水増し）、文書をまたいだ再主張は歓迎（それが裏付け）。

名前は見出し、本文ではない。subject/label/objectは1024バイトまで。長い知識は分解するか、 原文としてソースAPIに置く。weightは証拠の強さであって効果の大きさではない——「リスク2倍」は objectに書くか（喫煙 →リスク倍率→ 2倍）、原文に残す。

コンテキスト＝文脈

1つのコンテキストの中では、1つの綴りは1つの指示対象を意味する。

果物のAppleと会社のAppleは別のコンテキストに属する。この「一綴り一referent」の規約が、 グラフを曖昧性解消の装置ではなく知識の置き場として単純に保つ。 どのコンテキストに聞くかを選ぶのは言語理解の仕事、つまりLLMの仕事—— そのために GET /contexts は人間が書いた説明文、機械集計の統計（連想数・主要概念・ラベル見本）、 使用統計（読み・空振り・書き・最終アクセス）を1行に並べて返す。 「誰にも選ばれないコンテキスト」と「選ばれるのに何も答えないコンテキスト」はそこで見分けられる。

グラフは索引、アーカイブではない

骨格はグラフに、肉付けは原文に。

文書をインジェストするときは、事実の骨格を連想に落とすと同時に、原文そのものをソースAPIに登録する （ソースid → パッセージ）。サーバーは本文を空行境界で段落に分割して保持し、 すべての連想のattributionは source と paragraph でその段落を指せる—— citations APIで元の一文まで手繰って裏取りできるのはこの構造のおかげ。 段落に section（見出し）を付けておけば、グラフの読み出し結果のattributionに 解決済みのセクション名が同乗してくる。

検索ループ

接続したLLMが /protocol から受け取る規律。人間がcurlで辿っても同じ形になる。

01選ぶlist_contexts で対象の文脈を選定
02解決するresolve / resolve_label でcueを正規名に
03見取るdescribe→query で絞り込む
04広げるactivate / explore で関連知識を収集
05裏取りするcite_passage で原文を確認
06網を張るグラフで届かなければ search_passages

入口の正規化——cueはそのままでは照合しない

resolve の入口はNFKC正規化・大文字小文字の畳み込み・カタカナ→ひらがな変換の上に バイグラム転置索引を張ったもの。全半角の揺れ、かな表記の揺れ、軽い誤字はここで吸収される。 外れたら言い換えるか、dice_floor（既定0.3）を一時的に下げて曖昧照合を広げる。

候補には kind が付く。exact／alias は「cueが登録済みの綴りそのもの」、 containment／fuzzy は「文字列として重なっているだけ」。後者の高スコアは そっくりさんの可能性がある——京都は東京都に0.67で重なる。上位候補に付いてくる gloss（名前＋最重量の事実の要約）を読んでから採用する。スコアだけで採らない。

セマンティック層は「弱い入口を補う層」

埋め込みが設定されていれば、字句候補が無い・弱い（最高スコア < 0.5）ときに tier:"semantic" の候補が合流する。名前はグロスとして埋め込まれているので、 言い換え（醸造責任者→杜氏）や質問形のcueも着地する。tierを跨いだスコア比較はしない—— 字句のDice係数とコサイン類似度は別の物差しだ。床（semantic_floor）は 埋め込みモデルの属性であり、モデルを替えたら校正し直す。

テキストレーン——三つ組にならない知識の網

手順の細部、条件、数量、談話。骨格化できない知識は最初から文章のまま探す。

sources/search は段落単位の検索で、BM25の字句レーンと（TAGURU_EMBED_PASSAGES が 有効なら）段落埋め込みのセマンティックレーンを融合する。ヒットには lanes が付き、 どのレーンが何位で当てたかを常に申告する——vectorだけのヒットは言い換えが利いた印、 BM25だけのヒットは字面が合った印。どちらも証拠であって判定ではない。本文を読む。

質問ではなく、答えの形で検索する。多くの埋め込みモデルは質問文（「SSOはどのプランで使える?」）を、 その答えの文（「SSOはEnterpriseプランで利用できる。」）より遠くに置く。ありそうな平叙文を推測して その文で検索する——推測が当たっている必要はなく、探している文と同じ形であればいい。 逆向きの対策がdoc2query：インジェスト時に段落へ「この段落が答えになる質問」を添えておくと、 質問形の検索が質問側の索引に着地する。

インジェストの規律

検索の質はここで決まる。/protocol のインジェスト・ループの骨子。

  • 鋳造の前に照合（check before mint）：綴りを新しく作る前に resolve／resolve_label。あるものを再利用する。ラベル語彙は GET /contexts/{name}/labels が一覧する。
  • 暗黙のメンバーシップを明示する：「杜氏は高瀬」——誰の杜氏か。文書の主要エンティティから unreachable_from を叩けば、繋ぎ忘れた島が見つかる。
  • 文書が更新されたらdiff同期：sources/retract で旧版の寄与（weight・attribution・パッセージ）を撤回してから、新版を普通にインジェスト。概念とエッジは残り、weightだけが下がる。
  • 節目に語彙監査：vocabulary/audit が綴りの分岐候補を挙げる——字句の双子（表記ゆれ）と意味の双子（同義語の分裂）。候補であって判定ではない。同一referentなら正準を決めてエイリアスを張る。別物なのに衝突し続ける対（東京都/京都、前株/後株）は「別物」という事実として書く：(株式会社青嶺, 別物, 青嶺株式会社, 1.0)——以後、両者のglossが resolve の場で警告してくれる。
  • エイリアスは入口専用：結果は常に正準の綴りで返る。既存の2概念を繋ぐことはできない（それはマージであり、再構築の領分）。登録間違いは同じパスのDELETEで撤回する。

表現パターン

新しい機構は無い——メンバーシップのエッジと負のweightと同じ道具立てで表す。

手順（順序のある知識）

[{"subject":"日本酒の醸造","label":"最初の工程","object":"洗米","weight":1.0,"source":"工程書"},
 {"subject":"洗米","label":"次の工程","object":"浸漬","weight":1.0,"source":"工程書"},
 {"subject":"日本酒の醸造","label":"工程","object":"洗米","weight":1.0,"source":"工程書"}]
  • 順序は 次の工程 の連鎖（正準ラベルを1つに固定。分岐は複数の次の工程エッジ＝DAG）。開始は 最初の工程。所属（工程）が全ステップをハブに繋ぎ、網羅性の監査を効かせる。
  • 再生は「開始を引く→query {label:"次の工程"} で全対を取って並べる」。explore のホップ距離を順序に使わない——所属エッジがハブ経由の近道を作り、鎖上の位置と食い違う。
  • 分量・条件・コツはグラフに押し込まず原文に残し、sources/search で拾う。

因果

[{"subject":"ストレス","label":"引き起こす","object":"不眠","weight":1.0,"source":"論文A"},
 {"subject":"カフェイン","label":"引き起こす","object":"不眠","weight":-0.8,"source":"論文C"},
 {"subject":"運動","label":"防ぐ","object":"不眠","weight":1.0,"source":"論文D"}]
  • 「なぜX?」は query {label:["引き起こす","高める","要因"], object:"X"}。連鎖は activate のpathで見える——システムは A→B→C を見せるが A→C を主張しない。推移は読む側の判断。
  • 「引き起こさない」は因果ラベルに負のweight、「防ぐ」は独立した正のラベル。混ぜない。相関は相関するのまま、因果に昇格させない。
  • 議論が割れた因果は、1本のエッジ上の小さな正味weightと割れたattributionsとして現れる。答えでもそう言う。

ソースコード

  • 概念は識別子の正確な綴り、短い名前で：fetch_block であって CacheStore::fetch_block ではない——修飾は入口のスコアを薄める。名前空間はエッジで（defined_in → src/store.rs）。ファイルもパスも概念。
  • ラベル語彙は小さく決めて使い回す：kind・defined_in・calls・field・variant・returns・invariant・purpose でコードの事実の大半が書ける。
  • 大小文字の双子（型Frameとアクセサframe）は別概念で、resolveは両方を1.0で返す——同点は describe で見分ける。決してマージしない。
  • 入口が camelCase/snake_case/誤字を吸収する（fetchBlock も evict_cld も届く）。自然言語からの入口は識別子へのエイリアスで（退避ループ → evict_cold）。
  • いちばん価値が高いのはgrepが答えられない事実——不変条件・目的・設計上の制約を、識別子の上のエッジとして書く。
