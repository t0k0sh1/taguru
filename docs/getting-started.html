getting started

はじめる

インストールから最初の記憶、まとまった知識の投入、LLMエージェントとの接続、 そして運用の基本まで。ここを一往復すれば、Taguruを実際の作業に組み込める。

1. インストール

ひとつのcrateからサーバーとMCPブリッジの両方が入る。

cargo install taguru        # `taguru`（サーバー）と `taguru-mcp`（MCPブリッジ）

# ソースから
git clone https://github.com/t0k0sh1/taguru && cd taguru
cargo run --release

# Docker（linux/amd64 + linux/arm64）
docker run -d --name taguru \
  -p 127.0.0.1:8248:8248 \
  -v taguru-data:/data \
  ghcr.io/t0k0sh1/taguru:latest

2. 起動と設定

既定は 127.0.0.1:8248（電話キーパッドの TAGU。近隣ツールの既定ポートを避けて選んである）、 データは ./data。設定はすべて環境変数で、taguru --help が全変数を列挙する。 

taguru                       # そのまま起動
taguru --config taguru.env   # KEY=VALUE ファイルから読む —
                             # `docker run --env-file` と同じ方言なので1ファイルで両方を駆動できる。
                             # 実環境変数がファイルより優先。未知の TAGURU_* はtypoとして警告される
taguru version

3. 最初の記憶

コンテキストを作り、連想を入れ、手繰る。HTTPだけで一往復できる。

curl -X PUT localhost:8248/contexts/sake -H 'Content-Type: application/json' \
  -d '{"description":"青嶺酒造という架空の酒蔵の知識"}'

curl -X POST localhost:8248/contexts/sake/associations -H 'Content-Type: application/json' \
  -d '[{"subject":"青嶺酒造","label":"代表銘柄","object":"青嶺","weight":1.0,"source":"第1段落"}]'

curl -X POST localhost:8248/contexts/sake/activate -H 'Content-Type: application/json' \
  -d '{"origins":["青嶺酒造"]}'

activate の応答は強さ順の連想リストで、各行に strength（この呼び出し内での順位づけ）、 path（どう手繰ってきたか）、出典の attributions が付く。 エンドポイント全一覧と規律書は、起動中のサーバーの GET /protocol がそのまま配る。

4. Dockerで運用する

公開イメージはscratch上のサーバー単体——シェルもlibcも無い、パッチ対象の無い〜13 MB。

  • コンテナ内では 0.0.0.0:8248 にバインドする（loopbackだと -p の外から届かない）。つまり認証なしモードは公開した所まで届く——-p を 127.0.0.1 に絞るか、TAGURU_API_TOKEN を設定する。
  • 設定は同じ環境変数（-e、または --env-file taguru.env——--config が読むのと同一ファイル）。
  • プロセスは uid 65532 で走る。名前付きボリュームは初回に所有権を自動継承、バインドマウントは先に chown -R 65532（または --user "$(id -u)"）。
  • --read-only で動く（書き込み先は /data のみ）。docker stop は graceful shutdown（flush＋使用統計の書き出し）。
  • HEALTHCHECKは taguru health——バイナリが自分自身を突く（scratchにcurlは無い）。
# バックアップの検証もローカルツールチェーン不要
docker run --rm -v taguru-data:/data ghcr.io/t0k0sh1/taguru inspect /data

5. まとめて入れる — taguru import

初期ロードや移行はHTTPを介さない。JSONLのバッチファイルを、サーバーと同じWAL経由の書き込み経路で データディレクトリに直接適用する。 

taguru import batches/            # 配下の *.jsonl を名前順に
taguru import --dry-run batches/  # 検証と報告だけ。何も書かない

1ファイル＝1ソースの完全な真実が唯一のルール。適用は「そのソースを撤回してから、ファイルを適用」なので——

  • 冪等：同じファイルを2度importしても同じ状態。weightは二重加算されない。
  • 改訂可能：修正版ファイルは古い事実を丸ごと置き換える（エージェントがライブで行うdiff同期と同じ）。
  • 再試行可能：途中で失敗したファイルは直して再import——撤回が再試行を正確にする。
# バッチファイル：先頭がヘッダー、以降は1行1操作
{"taguru_batch": 1, "context": "sake", "source": "docs/aomine.md", "create": {"description": "酒蔵の知識"}}
{"passage": "青嶺酒造は1907年創業。杜氏は高瀬。"}
{"paragraph": 0, "section": "沿革"}
{"paragraph": 0, "question": "青嶺酒造の酒造りの責任者は誰?"}
{"subject": "青嶺酒造", "label": "杜氏", "object": "高瀬", "weight": 2.0}
{"alias": "Aomine Brewery", "canonical": "青嶺酒造", "kind": "concept"}

検証は適用と別パス：1行でも不正ならファイル全体が行番号付きで拒否され、何も書かれない。 稼働中のサーバーには同じ契約が POST /import にある（1リクエスト＝1バッチファイル）—— ダウンタイム無しの一括投入はこちら。 

# -d は改行を落とす。改行がフォーマットなので --data-binary で
curl -X POST localhost:8248/import -H 'Authorization: Bearer <key>' \
  --data-binary @docs-aomine.jsonl

行タイプの完全な意味論、上限の表、並行性の規則までの全契約は バッチ投入のリファレンスに。

文書からバッチファイルを作る — taguru extract

taguru extract は .md/.txt をOpenAI互換のチャットモデルに読ませ、 /protocol の規律で連想に分解して、文書ごとに1バッチファイルを書き出すオフラインの生産者。 サーバーは相変わらずモデルの資格情報を持たない——TAGURU_EXTRACT_* は抽出プロセスの環境だけに居る。 

TAGURU_EXTRACT_URL=https://api.openai.com/v1/chat/completions \
TAGURU_EXTRACT_MODEL=gpt-4.1 TAGURU_EXTRACT_API_KEY=$KEY \
taguru extract --context sake --description "酒蔵の知識" --out batches/ docs/
taguru import batches/
  • --out のマニフェストが「内容×モデル×プロンプト版×コンテキスト」を記録し、変わっていない文書はスキップ（--force で強制）。夜間の extract→import は変更分のモデル呼び出しだけで回る。
  • --questions N でdoc2query：段落ごとに検索質問を最大N件同梱。TAGURU_EMBED_PASSAGES のサーバーでは質問が段落の隣に埋め込まれ、質問形の検索が答えの形の本文に着地する。
  • モデル出力は信用せず、こちら側で契約を強制する——不正項目は捨ててカウントし、書き出す前にimportのパーサで再検証する。extractは「importが拒否するファイル」を作れない。
  • ローカルモデル（Ollama等）で回すなら：思考モードはOFF（reasoningトークンが時間予算を食い潰す）、コンテキスト窓を実サイズに（num_ctx 16384——4kのままだと静かに切り詰められて品質だけ落ちる）、タイムアウトはハードウェアに合わせTAGURU_EXTRACT_TIMEOUT_SECS、マニフェストはモデル名を信じるのでエイリアスの差し替え後は--force。

プロンプトが何を頼み何を強制するか、チャンク分割、マニフェストの詳細、信頼モデルと モデル級別の品質実測は文書抽出のリファレンスに。

6. LLMエージェントから繋ぐ — MCP

分解と組み立てはエージェントの仕事。規律はツール定義とMCPのinstructions（/protocolの全文）に 同梱されて自動で届くので、接続した時点でLLMは正しい手順を知っている。

ローカル（stdioブリッジ）

claude mcp add taguru -e TAGURU_URL=http://127.0.0.1:8248 -- taguru-mcp

これだけで「このフォルダの文書をsakeコンテキストに取り込んで」「青嶺酒造について出典付きで教えて」が、 ディレクトリ選択 → resolve → describe/query/activate → 引用、のループとして動く。 実際の一往復はウォークスルーで追える。

リモート（Streamable HTTP）

claude mcp add --transport http taguru https://your-host/mcp \
  --header "Authorization: Bearer $TAGURU_API_TOKEN"
# Claude API: mcp_servers = [{type: "url", url: "https://your-host/mcp",
#                             name: "taguru", authorization_token: "…"}]

POST /mcp はMCP Streamable HTTPトランスポート（ステートレス・プロファイル：素のJSON応答、 管理すべきセッション無し）を、他のAPIと同じBearerトークンの内側で話す。

claude.ai カスタムコネクタ（OAuth）

ヘッダーを貼れないクライアントには：TAGURU_PUBLIC_URL を設定し、コネクタを https://your-host/mcp に向け、同意ページで既存のAPIキーを貼る——以後その接続はそのキーとして 振る舞う（アクセスログでは key@client）。ディスカバリ、動的登録、PKCEは内蔵で、外部IdPは不要。 OAuthトークンが開くのは /mcp だけで、素のAPIはキーのみのまま。

localhostの外に出すならTLS必須（リバースプロキシで）。トークンは資格情報そのもの——持つ者が記憶を持つ。 クライアントごとに名前付きキーを切っておけば（TAGURU_API_TOKENS）、漏洩は1本の失効で済む。

7. 運用の基本

  • ヘルス：GET /health は書き込み経路が健全なら 200 ok、直近のflushが失敗していれば 503。ディスクが直れば1 flush間隔で自己回復する。
  • メトリクス：GET /metrics（Prometheusテキスト）。ルート別レイテンシ、キャッシュ/flush/WAL/埋め込みの結果、検索のヒット/空振り、500の原因内訳。コンテキスト名は意図的にラベルへ入れない（クライアントが名前を鋳造するため系列が無限に増える）——コンテキスト別の数字は GET /contexts の使用統計が持つ。
  • バックアップ：1コンテキスト＝1ファイルファミリー（.ctx .meta.json .passages.bin .passages.wal.jsonl .bm25.bin .vectors.bin .wal.jsonl）。必ず一括で。全ライターはfsync+renameなので、ファイルシステムのスナップショット（ZFS/Btrfs/LVM）はいつでも安全。稼働中の素朴な rsync/cp はファイル間の一貫性を保証しない——止めるか、本物のスナップショットを。POST /flush で直前に全dirtyを書き切れる。検証は taguru inspect /path/to/data（サーバーと同じ全検証ロード＋WALリプレイ。非ゼロ終了＝承認済みデータの破損）。
  • 排他：データディレクトリは同時に1プロセス（serve か import）。advisoryロック（flock系——ローカルディスクでは頼れるが、NFS/EFS/FUSEでは両側に許可され得る）。ローリングデプロイは新旧が重なった時点で新側が正しく退場する＝共有ボリュームでは stop-then-start（Kubernetesなら strategy: Recreate）。
