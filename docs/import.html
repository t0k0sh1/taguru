バッチ投入の契約——taguru import と POST /import

オフラインCLIと稼働中サーバーのエンドポイントが共有する、ひとつのファイル契約の完全なリファレンス。 初期ロード、インスタンス間の移行、抽出パイプラインの再生はここを通る。 ライブで少しずつ書くのは相変わらずHTTP API／MCPツールの仕事。

CLIの形

taguru import はバッチファイルをデータディレクトリへ直接適用する——稼働中のサーバー無しで、 そして決してサーバーの横では動かない：ディレクトリロックが相互に拒否する （advisoryなflock。ローカルディスクでは頼れるが、ネットワークファイルシステムでは両側に許可され得る）。 

taguru import batch/                  # batch/ 配下の全 *.jsonl を名前順に
taguru import --dry-run batch/        # 検証して報告する。何にも触れない
taguru import --config prod.env a.jsonl b.jsonl

終了コード：0 すべて適用 · 1 何かが拒否または失敗（詳細はstderr） · 2 使い方の誤り。

1ファイル＝1ソースの完全な真実

ファイルの適用とは：そのソースを撤回してから、ファイルを適用する。この一つの規則が運用上の性質を全部くれる。

  • 冪等——同じファイルを2度importしても同じ状態に着地する。weightは決して二重にならない。
  • 改訂可能——同じソースの修正版ファイルは古い事実を丸ごと置き換える。エージェントがライブで行う差分同期（retract_source → 再インジェスト）と同じ操作。
  • 再試行可能——途中で失敗した（容量・ディスク）ファイルは、直して再importする。撤回が再試行を正確にする。

連想の行が自分の source フィールドを持てない（拒否される）のもこのため：ヘッダーが名乗らないソースは 撤回を生き延び、再importのたびに二重化してしまう。ヘッダーのsourceがファイル内の全連想に刻印される。

ヘッダーだけで操作行の無いファイルは純粋な撤回——「このソースはもう何も主張しない」。

鋭い角がひとつ。retract-then-apply の順序ゆえに、apply段が持続的に失敗すると（現実的なのは容量）、 再importが成功するまでそのソースは空のまま（旧事実は撤回済み、新事実は拒否済み）になる。 また、撤回はソースのattributionを解放するが、語彙を未登録には戻さない：一度鋳造された概念やラベルは コンテキストの生涯にわたってidを保持する。名前が回転し続ける改訂ファイルを再importし続けるパイプラインは、 コンテキストを単調に成長させる。taguru inspect で数を、taguru estimate で余裕を監視し、 溜まった死重は taguru compact（オフライン）または POST /contexts/{name}/compact（稼働中・admin）が回収する——生きた事実・count・段落ロケータは 全て保存され、撤回済みレコードとアリーナの残骸だけが消える。

ファイル形式

JSON Lines、UTF-8。空行は無視。最初の非空行がヘッダー、以降の各行が1操作。

{"taguru_batch": 1, "context": "sake", "source": "docs/aomine.md", "create": {"description": "酒蔵の知識"}}
{"passage": "青嶺酒造は1907年創業。杜氏は高瀬。"}
{"paragraph": 0, "question": "青嶺酒造の酒造りの責任者は誰?"}
{"paragraph": 0, "section": "沿革"}
{"subject": "青嶺酒造", "label": "創業年", "object": "1907年", "weight": 1.0}
{"subject": "青嶺酒造", "label": "杜氏", "object": "高瀬", "weight": 2.0}
{"alias": "Aomine Brewery", "canonical": "青嶺酒造", "kind": "concept"}
{"alias": "設立年", "canonical": "創業年", "kind": "label"}

Header（ヘッダー行）

taguru_batch は 1 でなければならない（このビルドが読む唯一の版）。 context と source は必須。create は任意で、 PUT /contexts/{name} と同じフィールド（description・pinned・ dice_floor・semantic_floor）を運ぶ——使われるのはコンテキストが まだ存在しないときだけ。createブロック無しで不在のコンテキストへimportするのは拒否。

Association（連想行）

subject / label / object / weight—— associationsエンドポイントのフィールドから source を除いたそのもの（理由は上記）。

Alias（別名行）

alias / canonical / kind（"concept" か "label"）。 エイリアスはコンテキストレベルで、ソーススコープではない：再importは再登録（不変ペアはno-op）、 撤回はエイリアスを消さない。canonicalがこのファイルの連想で初めて導入される名前でもよい——連想が先に適用される。 既存のエイリアスを別のcanonicalへ付け替えるのは競合で、ファイルを失敗させる——importが黙って語彙を 配線し直してはならない。直し方は意図的に一手間：旧登録を撤回してから （DELETE /contexts/{name}/aliases に正確な綴りで、またはMCPの remove_aliases）、 再importする——撤回が再試行を正確にする。バッチファイル自身はエイリアスを消せない。これも故意。

Passage（原文行）

そのソースの原文テキスト。1ファイルに最大1つ。同じソースidの背後に格納され、 sources/lookup と sources/search が提供する。

Question（doc2query行）

paragraph / question：このファイルのpassageの1段落に付く検索質問。 paragraph はサーバーの空行分割での0起点位置（passage行の無いファイルにquestion行があれば拒否）。 1行1問、同じ段落に複数可（上限まで）。TAGURU_EMBED_PASSAGES のサーバーでは各質問が段落の隣に 埋め込まれ、質問形の検索が答えの形の本文に着地する。passageの分割に存在しない段落を名指した質問は 格納時に捨てられ、応答で数えられる（questions_dropped）——よくある原因は生産側の分割が サーバーの分割からドリフトすること。taguru extract --questions N は サーバー自身のsplitterを使うのでドリフトできない。

Section（見出し行）

paragraph / section：このファイルのpassageの中で、名前付きセクションが 始まる位置を示す見出し（0起点、Questionと同じ分割規約。passage行が無ければ拒否）。 セクションは自分の段落から、次のセクションの開始または本文末尾まで暗黙に延びる——「終わり」の行は無い。 分割に存在しない段落を名指したセクションは格納時に捨てられ数えられる（sections_dropped、 原因の相場もquestionsと同じ）。attributionが一致する段落ロケータを持つ連想は、そのattribution上で セクションを報告する：recall・query・explore・activate・ unreachable_from のすべてが attributions[].section を解決し、ロケータがどの セクションにも入らなければ null。POST /contexts/{name}/citations も同じ解決を その1段落について行う。passage検索のヒットはこのフィールドをまだ読まず、格納内容に関わらず省略し続ける—— 別途のフォローアップ。

未知のフィールドと認識できない行形状は、行番号付きで拒否される。

上限（APIと同じもの。行ごとに強制）

対象上限
context名64バイト
source id・subject/label/object・alias/canonical1,024バイト
create.description4,096バイト
weight有限、|w| ≤ 1e6
passage8 MiB（HTTP既定のボディ上限）
question 1件512バイト
1段落あたりのquestion数8
section 1件512バイト
任意の1行16 MiB

ファイル自体に操作数の上限は無い。適用は内部でAPIのバッチサイズ（10,000）ごとに刻まれ、 長い実行は定期的にflushするので、どのコンテキストのWALも TAGURU_WAL_MAX_BYTES に近づかない。

検証、それから適用

  • すべてのファイルがparseでき、かつどの2ファイルも同じ (context, source) を主張しない——それを確かめてから初めて何かが適用される。1行の不正が実行全体を行番号付きで拒否し、何も書かれない（--dry-run は設計どおりそこで止まる）。
  • apply段でしか発見できない失敗（容量・ディスク）はファイルごとに報告され、残りのファイルは続行する：ファイルは構造上独立（1ファイル1ソース）で、部分適用されたファイルは再importで治る。
  • 書き込みはサーバーと同じレジストリを通る——WALに載せてfsync、予算を強制、最後にflush。だから taguru inspect も次回のbootも、ライブのインジェストが生んだのと寸分違わぬ状態を見る。importは各コンテキストの使用統計でwriteとして数えられる。

先に全部を検証することには費用がある：parse済みの全ファイルが適用までメモリに留まるので、 実行のフットプリントはファイル群の合計サイズに比例する（最大の1つではなく）。 100万文書の移行は数回の呼び出しに分けて流す（ディレクトリを輪切りに）——importは冪等で 各ファイルが自分のソースを運ぶから、分割は何も費やさない。

埋め込み

TAGURU_EMBED_URL／TAGURU_EMBED_MODEL が設定されていれば（--config 経由でも）、 触れたすべてのコンテキストが実行の最後に変更されたグロスを再埋め込みする——サーバーの TAGURU_EMBED_AUTO と同じ冪等なrefresh。--no-embed でスキップ。 refreshの失敗は何も巻き戻さない：グラフはimport済みで耐久であり、実行は回復手順 （稼働サーバーの POST /contexts/{name}/embeddings/refresh）を名指しして1で終了する。

同じ契約をHTTPで：POST /import

リクエストボディが1つのバッチファイルであり、1リクエストが1ソースの完全な真実を述べる。 稼働中のシステムはダウンタイム無しで一括投入できる。 

# --data-binary であって -d ではない：curlの -d は改行を剥ぎ、改行こそがフォーマット
curl -X POST localhost:8248/import \
  -H 'Authorization: Bearer <key>' \
  --data-binary @docs-aomine.jsonl
# → {"result": {"context": "sake", "source": "docs/aomine.md",
#    "created": false, "retracted": 12, "associations": 14,
#    "aliases": 2, "passage_stored": true}, "status": "ok", …}
  • 先に検証（不正な行は行番号付きの400で、何も適用されない）、それからretract-then-apply——同じバッチの再POSTは、CLIの再実行と同じくソースを正確に置き換える。応答はCLIが印字するのと同じ数字を運ぶ。
  • どのエンドポイントも従うものにこれも従う：Bearer認証、ボディ上限（TAGURU_MAX_BODY_BYTES、既定8 MiB——それより大きいバッチはオフラインCLIか上限の引き上げ）、リクエストタイムアウト、キーごとのレート制限。
  • ステータスはAPIの流儀：400 不正なバッチ（行番号付き） · 404 コンテキスト不在でcreateブロック無し · 409／507 部分書き込み（競合／容量。何が着地したかはメッセージが名指しし、撤回が修正後の再試行を正確にする） · 500 WALまたはディスクが拒んだ書き込み。
  • 埋め込みはリクエスト内ではrefreshされない。TAGURU_EMBED_AUTO が有効なら、ライブの書き込みと同じく次のflushに同乗する。

異なるソースの2バッチは並行してPOSTしてよい——バッチ内の各書き込みはコンテキスト上で直列化される。 同じソースの2つのimportを競走させてはならない：一方のretractとapplyの間にもう一方が割り込み、 両バッチの和集合が生き残り得る。1ソース、同時に1ライター。

入口の選び方：サーバーがまだ無い、または大量の初期ロード → taguru import （リクエスト単位の上限なし、flushは最後に1回）。答え続けなければならないライブ系 → POST /import。

バッチのストリームと taguru export（バックアップ／復元）

1ファイル（1リクエストボディ）は、1バッチまたは連結された複数バッチを運べる—— 各 taguru_batch ヘッダー行が前のバッチを閉じ、次を開く。バッチごとの 検証・retract-then-apply・(context, source) の一意性は単体ファイルと同一。

そのストリームを書き出すのが taguru export --out DIR [CONTEXT...]（オフライン、 ディレクトリロック下）と、稼働中サーバーの GET /contexts/{name}/export （ボディがそのままJSON Linesのストリーム）。1コンテキスト＝1ストリーム：ソースid順に1バッチずつ、 先頭バッチに create ブロック（description・pinned・floors）、 末尾バッチに別名表が同乗する。復元は通常のimport——ソース単位の置き換えなので冪等で、 スナップショットと厳密に一致させたいときはコンテキストを先に削除してから流し込む。

  • 忠実度：同一ソースがn回主張した事実はn本の連想行として再現される（corroborationの count が往復を生き延びる）。段落ロケータは初回主張行に載る（first-write-winsの再現）。
  • ソース無しの重み（APIの source 省略で入ったもの）は予約ソース export:unsourced のバッチに載る——数字は保存され、出所はattributionが語る。 実ソースがこの予約idと衝突した場合、exportは黙って混ぜずに拒否する。
  • 完全撤回済みのエッジ（weight 0）は排出されない：export → import の往復は、 追記専用イメージが抱える死んだレコードを削ぎ落とす——オフラインのコンパクションを兼ねる。
  • 正準名が生きた連想を1本も持たない別名は落とされ、件数が報告される（importの契約上、 正準名は連想行によって鋳造されるため）。
  • 複数バッチのストリームへの POST /import は {"batches": [バッチごとの結果…]} で答える（単体バッチは従来の形のまま）。途中で拒否されたら手前のバッチは着地済み—— 各バッチがretract-then-applyなので、直したストリームを丸ごと再POSTすれば正確に収束する。
  • レンダリングは決定的：同じ状態は同じバイト列に書き出され、バックアップ同士がdiffできる。
# 稼働中サーバーから1コンテキストを退避し、別サーバーへ復元する
curl -H 'Authorization: Bearer <key>' \
  localhost:8248/contexts/sake/export > sake.jsonl
curl -X POST other-host:8248/import \
  -H 'Authorization: Bearer <key>' --data-binary @sake.jsonl
# オフラインなら： taguru export --out backups/ ; taguru import backups/

バッチファイルを生産する

Taguruのサーバーは散文から事実を抽出しない——それは読む側のLLMの仕事で、サーバーは決して モデルの資格情報を持たない。生産者（producer）が文書を読み、モデルに /protocol の規律で 連想を吐かせ、文書idをソースとして文書ごとに1バッチファイルを書く。

そういう生産者がひとつ同梱されている：taguru extract—— 上記そのものを任意のOpenAI互換チャットエンドポイントで行うオフラインのサブコマンドで、 TAGURU_EXTRACT_* の資格情報を自分の環境だけに持つ。カスタムパイプラインも第一級のまま： BedrockガイドのConverseループと /protocol のツール語彙は そういう生産者の両半分であり、バッチファイルはそのオフライン直列化にすぎない。
