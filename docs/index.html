手繰る（たぐる）——
糸口から知識を引き出す、LLMのための長期記憶

Taguruは知識を (subject, label, object, weight, source) の連想として蓄積し、質問が「何について」かという糸口（cue）から連想グラフを 手繰って知識を引き出す記憶サーバー。ベクトル類似度は検索の入口ではなく、 証拠を運ぶ一本のレーンとして併走する。

License: MIT 実装: Rust・単一バイナリ（~13 MB static） インターフェイス: HTTP API / MCP（stdio・Streamable HTTP）

何をするサーバーか

保持するもの、検索のしかた、そして仕事の分担。

「青嶺酒造の杜氏は高瀬」という知識は (青嶺酒造, 杜氏, 高瀬, 1.0, docs/aomine.md) という一本の連想になる。weightは符号付きで、負の値は「〜ではない」という否定の主張。 同じ三つ組を別の文書が再主張すればweightが加算され、出典ごとのattributionが残る—— 2つの情報源が独立に支持する事実と、1つの情報源が強く主張する事実は、最後まで区別できる。

検索はまず構造から。cueを正規化された照合（表記ゆれ・かな/カナ・軽い誤字を吸収）で グラフ上の概念に解決し、そこから連想を手繰る——これが第一レーン。 元の文章は段落単位のテキスト検索（BM25＋設定時は埋め込み）でもう一本のレーンとして引ける。 三つ組に収まらなかった手順の細部や条件は、こちらでしか見つからない。

言語を理解する仕事——文書を事実に分解する、コンテキストを選ぶ、結果を文章に組み立て直す——は クライアント（LLM）の仕事。サーバーは構造の保管と探索だけを引き受け、 そのための規律書は GET /protocol でサーバー自身が配布する。

 ウォークスルー：LLMが知識を引き出すまで

実際のMCPリクエストとレスポンスで追う一往復。グラフレーンで答える質問と、テキストレーンに切り替える質問。

30秒で試す

サーバーを起き、コンテキストを作り、事実を一本入れて、手繰ってみる。

# インストール（server `taguru` と MCPブリッジ `taguru-mcp` が入る）
cargo install taguru
taguru   # 127.0.0.1:8248 で待ち受け（データは ./data）

# コンテキストを作る
curl -X PUT localhost:8248/contexts/sake -H 'Content-Type: application/json' \
  -d '{"description":"青嶺酒造という架空の酒蔵の知識"}'

# 連想を一本入れる
curl -X POST localhost:8248/contexts/sake/associations -H 'Content-Type: application/json' \
  -d '[{"subject":"青嶺酒造","label":"代表銘柄","object":"青嶺","weight":1.0,"source":"第1段落"}]'

# 手繰る
curl -X POST localhost:8248/contexts/sake/activate -H 'Content-Type: application/json' \
  -d '{"origins":["青嶺酒造"]}'

エンドポイントの一覧と、インジェスト／検索の規律の全文は、起動中のサーバーの GET /protocol が配布する——LLMクライアントが受け取るものと同じ一次資料。 続きははじめるへ。

設計原則

一

構造が本線、類似度は一レーン

入口は正規化された照合と概念解決。埋め込みは設定されていれば「弱い入口を補う層」と「テキストレーンの並走」として効く。どのレーンが当てたかは結果のlanesやtierが常に申告する。

二

クライアントはLLM

文書の分解も、答えの組み立ても、コンテキスト選びも言語理解の仕事なのでLLM側。サーバーは規律書（/protocol）とツール定義を自ら配り、接続した瞬間からLLMが正しい順番でツールを呼べるようにする。

三

答えは原文まで手繰れる

グラフはアーカイブではなく索引。すべての事実は出典付きで、attributionのsourceとparagraphから元の段落そのものを引用（citation）で裏取りできる。

四

200を返した書き込みは失われない

受理された書き込みはWALにfsyncしてから適用される。クラッシュしても再起動時のリプレイで戻る。ディスクが真実、メモリはコンテキスト単位のキャッシュにすぎない。
