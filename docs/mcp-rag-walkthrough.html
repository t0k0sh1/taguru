taguru mcp · retrieval walkthrough

手繰る（たぐる）——
Taguru MCPでLLMが知識を引き出すまで

AIチャットボットがユーザーの質問に答えるとき、裏側でLLMはTaguru MCPのツールを何度も呼び出し、 構造化された知識グラフと元の文章を手繰り寄せてから返答している。 その一往復を、実際に動かしたサーバーへの本物のリクエストとレスポンスで追う。

実行方法：ローカルのTaguruサーバーへ POST /mcp（Streamable HTTP） 構成：埋め込み未設定＝字句レーンのみ 題材：架空の酒蔵「青嶺酒造」

まず全体像

Taguru MCPが何を保持し、どう検索するかの最小限の説明。

Taguruは知識を (subject, label, object, weight, source) の連想（アソシエーション）として溜め込む、LLM向けの長期記憶サーバーだ。 「青嶺酒造の杜氏は高瀬」であれば (青嶺酒造, 杜氏, 高瀬, 1.0, docs/aomine.md) という一本の連想になる。

検索はベクトル類似度から始めない。まず「何についての質問か」という手がかり（cue）から概念名を解決し、そこを起点に連想のグラフを辿る——これが「手繰る」の名の由来であり、第一の検索レーンになる。 元の文章（パッセージ）はもう一本のレーンとして並走し、BM25と（設定されていれば）埋め込みの類似検索で段落ごとに引ける。三段論法や条件・手順のようにグラフの三つ組に収まらなかった知識は、こちらでしか見つからない。

知識を分解して連想に変換すること、検索結果を文章に組み立て直すことは、サーバーの仕事ではなくLLM＝MCPクライアントの仕事。サーバーは構造の保管と探索だけを引き受ける。

コンテキスト
list_contexts create_context update_context delete_context
書き込み（ingest）
add_associations store_passages lookup_passages list_sources
検索・読み出し

語彙・保守

全26ツールのうち、色付きの6つ（藍 がグラフレーン、朱 がテキストレーン）を、このあとの2つのシナリオで実際に呼び出す。

01選ぶlist_contexts で対象を選定
02解決するresolve / resolve_label でcueを正規名に
03見取るdescribe→query で絞り込む
04広げるactivate / explore で関連知識を収集
05裏取りするcite_passage で原文を確認
06網を張るグラフで届かなければ search_passages

シナリオ1：知っていることを聞かれる

グラフに構造化済みの事実で答えが組み立てられるケース。resolve → describe → activate → cite_passage と手繰っていく。

ユーザー

青嶺酒造について教えて。根拠も教えて。

接続直後、LLMは initialize の応答で instructions フィールドを受け取る。これは GET /protocol の全文そのもので、ここまでの「まず全体像」に書いた検索ループの規律が、会話が始まる前にLLMへ手渡されている。以降はその規律に沿ってツールを呼ぶだけでよい。
list_contexts GET /contexts

どのコンテキスト（＝どの「文脈」の知識ベース）を検索すべきか、まず一覧から選ぶ。

Request

tools/call
  name: "list_contexts"
  arguments: {}

Response

{
  "total": 1,
  "contexts": [{
    "name": "sake",
    "description": "青嶺酒造という架空の酒蔵の知識（沿革・杜氏・方針・醸造工程）",
    "stats": {
      "associations": 12,
      "top_concepts": [
        {"label": "青嶺酒造", "count": 6},
        {"label": "日本酒の醸造", "count": 5}
      ]
    }
  }]
}

人が書いた説明文と機械集計の統計が同じ行に並ぶ。LLMはこの一覧だけで「sakeコンテキストに聞けばよさそうだ」と判断できる。

resolve POST /contexts/sake/resolve

質問文から抜き出した「青嶺酒造」という手がかり（cue）を、グラフ上の正式な概念名に解決する。

Request

{
  "context": "sake",
  "cue": "青嶺酒造"
}

Response

[{
  "name": "青嶺酒造",
  "score": 1.0,
  "tier": "lexical",
  "kind": "exact",
  "gloss": "青嶺酒造。代表銘柄は青嶺。創業年は1907年。所在地は長野県。杜氏は高瀬。"
}, {
  "name": "青嶺",
  "score": 0.5,
  "kind": "containment",
  "gloss": "青嶺。青嶺酒造の代表銘柄は青嶺。"
}]

kind: "exact" はcueがそのまま登録済みの綴りだった印。glossにはその概念の主要な事実が要約されており、この時点で答えの下書きはほぼ見えている——ただしまだ裏取りしていない。2件目の「青嶺」は文字列として重なっているだけのcontainment候補で、スコア0.5と低く、正しく脇に残る。

describe POST /contexts/sake/describe

「青嶺酒造」という概念が、どんなラベルで何件の事実を持つかをまず見取る。プロフィール全体を勘で引く前の下見。

Request

{
  "context": "sake",
  "concept": "青嶺酒造"
}

Response

{
  "concept": "青嶺酒造",
  "as_subject": [
    {"label":"代表銘柄","count":1},
    {"label":"創業年","count":1},
    {"label":"所在地","count":1},
    {"label":"杜氏","count":1},
    {"label":"行う","count":1},
    {"label":"醸造工程","count":1}
  ],
  "as_object": []
}

6種のラベルが見え、「全部聞けば足りる」規模だと分かる（大きなハブ概念ならここでlabelを絞ってから query するところ）。行うラベルの存在は次のactivateで否定の事実として効いてくる。

activate POST /contexts/sake/activate

「青嶺酒造」を起点に拡散し、直接の事実と、そこから連なる関連知識を強さ順に集める。関連知識収集の主力ツール。

Request

{
  "context": "sake",
  "origins": ["青嶺酒造"]
}

Response（抜粋）

[
 {"strength":0.5,"path":["青嶺酒造"],
  "association":{"label":"杜氏",
   "object":"高瀬","weight":1.0,
   "attributions":[{"source":
    "docs/aomine.md","paragraph":1,
    "section":"概要"}]}},
 {"strength":0.5,"path":["青嶺酒造"],
  "association":{"label":"行う",
   "object":"大量生産","weight":-1.0,
   "attributions":[{"source":
    "docs/aomine.md","section":
    "経営方針"}]}},
 … 代表銘柄・創業年・所在地も strength 0.5 で同様に
 {"strength":0.0417,"path":
  ["青嶺酒造","日本酒の醸造"],
  "association":{"label":"最初の工程",
   "object":"洗米", …}}
 … 醸造工程の連鎖が2ホップ先まで、より弱く続く
]

直接つながる事実はstrength 0.5で並び、pathが1ホップの["青嶺酒造"]。醸造工程（別の連想の連鎖）はpathが2ホップに伸びるぶんstrengthが減衰して混ざり込む——「青嶺酒造は何をしているか」まで自然に広がる一方、遠い情報ほど弱く扱われる。負のweight: -1.0は「大量生産を行わない」という否定の事実で、そのままプラスの主張と読み違えないための符号。attributionsにはsectionまで解決済みで付いてくるので、この時点で「概要／経営方針のどちらに書かれているか」も分かる。

cite_passage POST /contexts/sake/citations

「杜氏は高瀬」の根拠を、要約や連想ではなく元の文章そのもので裏取りする。

Request

{
  "context": "sake",
  "source": "docs/aomine.md",
  "paragraph": 1
}

Response

{
  "text": "杜氏は高瀬が務めている。高瀬は先代から技術を受け継ぎ、伝統製法を守り続けている。",
  "source": "docs/aomine.md",
  "section": "概要"
}

paragraphにはactivateのattributionsに載っていたparagraph: 1の値をそのまま渡しただけ。連想は「杜氏＝高瀬」という骨格だけを持ち、「先代から技術を受け継ぎ」という肉付けは骨格化されずに原文側に残っている——だからこそ引用して初めて手に入る。

アシスタント

青嶺酒造は1907年に長野県で創業した蔵元で、代表銘柄は「青嶺」です。杜氏は高瀬さんが務めており、先代から技術を受け継いで伝統製法を守っています。大量生産は行わず、仕込み規模を抑えて一つひとつの工程に手間をかける方針とのことです。

出典：docs/aomine.md ／ 概要 「杜氏は高瀬が務めている。高瀬は先代から技術を受け継ぎ、伝統製法を守り続けている。」

シナリオ2：グラフに無い知識を聞かれる

三つ組に収まらない手順の細部（条件・数量）は、そもそもグラフに書かれていない。describeで空振りを確認してから、テキストレーンに切り替える。

ユーザー

浸漬はどのくらいの時間行いますか？

describe POST /contexts/sake/describe

resolveで「浸漬」を解決した後、まずグラフ側にどんな事実があるかを確認する。

Request

{
  "context": "sake",
  "concept": "浸漬"
}

Response

{
  "concept": "浸漬",
  "as_subject": [
    {"label":"次の工程","count":1}
  ],
  "as_object": [
    {"label":"次の工程","count":1},
    {"label":"工程","count":1}
  ]
}

「浸漬の次は麹造り」という順序はあっても、「何分」という数量のラベルは無い——手順の骨格は連想として書かれているが、条件や分量のような細部は最初から三つ組に押し込めず、文章側に残す設計だからだ。ここでグラフを追うのを打ち切り、テキストレーンに切り替える。

search_passages POST /contexts/sake/sources/search

質問文をそのまま投げず、あり得る答えの文に言い換えて検索する——「浸漬は何分？」ではなく「浸漬は夏場40分、冬場90分ほど行う」。当たる保証は要らない、探している文と同じ形であればよい。

Request

{
  "context": "sake",
  "query": "浸漬は夏場40分、冬場90分ほど行う",
  "limit": 3
}

Response

[{
  "source": "docs/process.md",
  "paragraph": 1,
  "score": 10.93,
  "text": "洗米の後は浸漬を行う。浸漬は米に適切な水分を吸わせる工程で、当蔵では夏場は40分、冬場は90分を目安に浸漬時間を調整している。気温や米の状態によって毎回微調整する。",
  "lanes": {
    "bm25": {"rank":1,"score":10.93}
  }
}, { // paragraph 2, score 1.02 … }]

lanesにbm25しか無いのは、このサーバーが埋め込み未設定＝字句レーンのみで動いているため（TAGURU_EMBED_URLを設定すればvectorレーンも並走し、言い換えにも強くなる）。今回は「浸漬」「40分」「90分」という語そのものが原文にあるので、BM25だけで一発当選している。

アシスタント

浸漬は、夏場は40分、冬場は90分を目安に行っているとのことです。ただし固定値ではなく、気温や米の状態に応じて毎回微調整するそうです。

出典：docs/process.md（paragraph 1） 「洗米の後は浸漬を行う。浸漬は米に適切な水分を吸わせる工程で、当蔵では夏場は40分、冬場は90分を目安に浸漬時間を調整している。気温や米の状態によって毎回微調整する。」

この質問はグラフの連想を1件も使わなかった。グラフが本線、テキスト検索は安全網——三つ組に落とせた事実はグラフから素早く正確に、落とせなかった細部はテキストレーンから拾う、という役割分担がそのまま表れている。

この往復から読み取れる設計

一

ベクトル類似度は入口ではない

resolveはまず正規化された文字列一致（表記ゆれ・かな/カナ・軽い誤字を吸収）で解決し、埋め込みが無ければそれで完結する。似ている文字列と同じものは別概念として扱われ、glossが両者の違いを教えてくれる。

二

答えは常に原文まで手繰れる

グラフ経由でもテキスト経由でも、最終的な一文はattributionsや検索結果のsource／paragraphからcite_passageで裏取りできる。今回の2つの最終回答は、どちらも要約ではなく実在する段落を引用して締めている。

三

グラフとテキストは役割が違う

「誰が」「いつ」のような骨格はグラフが速く正確に答え、「どのくらい」「どんな条件で」のような細部は文章のまま残る。LLMはdescribeで空振りを検知したら、質問文ではなく答えらしい一文に言い換えてsearch_passagesに切り替える。

四

規律はツール定義とinstructionsに同梱される

この一連の手順をLLMがどこかで学習している必要はない。initializeのinstructionsと各ツールの説明文自体が検索ループの規律を運んでくるので、接続した時点でLLMは正しい順番を知っている。
